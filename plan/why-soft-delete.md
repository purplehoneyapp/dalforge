Updating your **`dalforge`** tool to support a first-class `softdelete: true` flag is absolutely the best approach. 

Since you own the code generator, baking soft-delete directly into the Data Access Layer (DAL) is a massive architectural advantage. If you were to just add `deleted_at` as a standard column and handle it manually in your Go services, you would inevitably encounter a scenario where a developer (or you, six months from now) forgets to add `if user.DeletedAt != nil` and accidentally operates on a deleted account.

Here is why updating `dalforge` is the superior choice, along with exactly what the generator needs to do under the hood.

### Why a `softdelete: true` flag is better:

1. **Global Consistency (Safety):** Every `GetByID`, `GetByUid`, and `List` method generated by `dalforge` will automatically append `AND deleted_at IS NULL` to the SQL query. The service layer never even knows deleted records exist.
2. **Clean Service Layer:** Your `AuthService` and `SubscriptionService` stay purely focused on business logic without being cluttered by database state checks.
3. **Reusability:** You will likely need soft deletes for other entities later (like `Matches` or `Quests`). Building it into `dalforge` now solves it for the entire project.
4. **Transparent Deletions:** Your existing `Delete(ctx, entity)` interface method remains exactly the same, but the underlying implementation changes from a `DELETE` to an `UPDATE`.

### What `dalforge` needs to change in the generated code:

If you add `softdelete: true` to `user.yaml`, your generator should alter its output in these four ways:

#### 1. The Schema (`user.sql`)
The generator should automatically inject a `deleted_at` column:
```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    version INT DEFAULT 0,
    -- ... existing fields ...
    created TIMESTAMP,
    updated TIMESTAMP,
    deleted_at TIMESTAMP NULL -- 👈 Injected by dalforge
) ENGINE=InnoDB;
```

#### 2. The `Delete` Method (`user.gen.go`)
Instead of issuing a `DELETE FROM users WHERE id = ?`, the generator should swap the SQL inside the `delete` method:
```go
func (d *userRepository) delete(ctx context.Context, id int64) error {
	// ...
	query := `
		UPDATE users
		SET deleted_at = NOW(), updated = NOW(), version = version + 1
		WHERE id = ? AND deleted_at IS NULL
	`
	// ...
}
```

#### 3. The `Get` and `List` Methods (`user.gen.go`)
Every single `SELECT` query must be scoped to ignore deleted records:
```go
// Inside getByID, getByUid, getByDeviceId, etc.
query := `
    SELECT id, version, app_version, device_id, platform, timezone, uid, created, updated
    FROM users
    WHERE id = ? AND deleted_at IS NULL
`
```

#### 4. The Sweeper/Admin Methods (Optional but recommended)
Because the standard methods now hide deleted records, your backend cron jobs (like the 30-day sweeper we discussed) will need a way to find and permanently delete them. `dalforge` should generate two extra methods when `softdelete: true` is present:
* `HardDelete(ctx, entity)`: Issues the actual `DELETE FROM users WHERE id = ?`.
* `ListDeleted(ctx, limit)`: Returns records where `deleted_at IS NOT NULL`.

### ⚠️ The Unique Constraint Trap (Crucial for MySQL)

There is one major database trap you need to account for in your `dalforge` logic. 

Looking at your `user.sql`, you have:
```sql
CREATE UNIQUE INDEX idx_uid ON users (uid);
CREATE UNIQUE INDEX idx_device_id ON users (device_id);
```

If you soft-delete a "ghost" user, their `device_id` remains in the database. If that same device tries to hit `/v1/anonymous/register` again, **MySQL will throw a Unique Constraint Violation** because the `device_id` still exists (even though `deleted_at` is set).

**How `dalforge` should solve this:**
When `softdelete: true` is enabled, the `Delete` method shouldn't just set `deleted_at`. It should also "scramble" or `NULL` out the unique identifiers so the values are freed up for future rows. 

For example, your generated `delete` SQL could look like this:
```sql
UPDATE users
SET 
    deleted_at = NOW(),
    updated = NOW(),
    -- Scramble unique fields so the device can register again
    device_id = CONCAT(device_id, '-deleted-', UUID()), 
    uid = CONCAT(uid, '-deleted-', UUID()),
    version = version + 1
WHERE id = ? AND deleted_at IS NULL
```

By updating `dalforge` to handle the query scoping and the unique-constraint scrambling automatically, you create an incredibly resilient, enterprise-grade DAL!
